### 修复删除工作区时后端事件循环阻塞导致应用卡死

#### 问题描述

- 现象：删除包含大量文件的工作区时，桌面应用/前端出现"卡死"，按钮一直显示"删除中..."，期间其他 API 请求无响应。
- 原因：`WorkspaceRegistryService.delete_workspace` 在异步协程中直接执行 `shutil.rmtree`、`shutil.move` 等同步文件 IO 操作。删除大目录时会长时间阻塞 asyncio 事件循环，导致后端无法响应任何其他请求，前端表现为主进程冻结。

#### 改动内容

- 将 `delete_workspace` 改为异步方法，所有同步文件 IO（会话目录 detach/清理、工作区目录递归删除、`LocalIPythonBox.shutdown_kernel`、`AutoTaskStore.clear_workspace`）均通过 `asyncio.to_thread` 放到线程池执行。
- 修正此前 `agent_service.interrupt_session` 被调用但未被 `await` 的问题，使其真正生效。
- 更新 API 路由 `workspaces_core.delete_workspace` 以 `await` 调用新的异步服务方法。
- 同步更新单元测试，将同步调用改为 `await`。

#### 文件变更

- `apps/backend/app/services/workspace_registry.py`: `delete_workspace` 改为异步，文件 IO 走 `asyncio.to_thread`。
- `apps/backend/app/api/routes/workspaces_core.py`: 路由中 `await service.delete_workspace(...)`。
- `apps/backend/tests/test_session_delete_workspace_binding.py`: 测试改为异步并 `await`。

### 修复删除工作区后前端可能出现的请求风暴与页面冻结

#### 问题描述

- 现象：部分场景下删除当前工作区后，前端连续发起多次 `GET /api/workspaces?summary_only=true`，界面进入无响应状态。
- 原因：`useTaskWorkspaces.loadWorkspaces` 缺少并发/短时重复调用保护，且 `useWorkspaceRouteSync` 在删除后切换 fallback 时存在重入窗口，极端情况下会触发状态震荡。

#### 改动内容

- 为 `useTaskWorkspaces.loadWorkspaces` 增加“同请求在飞返回同一 Promise”和“150ms 内重复触发直接返回缓存”两层保护，避免工作区列表接口被连续请求。
- 为 `useWorkspaceRouteSync` 增加同步锁与“最后导航 URL”去重，防止删除后的 fallback 路由调整进入循环导航。
- 优化新建工作区时 Node.js 运行环境初始化失败的提示：当错误与 fnm/Node.js 相关时，给出安装 fnm 或取消勾选 Node.js 的明确引导；在新建对话框的 Node.js 选项上补充“需要本地已安装 fnm”说明。
